Jeeves Skills for the Knowledge Base

Three new skills form the ingestion pipeline. Three existing skills get updates. Together they implement Karpathy's LLM ingest pattern on your Obsidian vault.

New Skill: vault-audit

Purpose & Trigger

What it does: Walks your entire Obsidian vault and produces a structured report of its contents, conventions, and cross-linking state. This is the foundation skill — everything else depends on understanding what the vault actually looks like.

When to use:

  • First time: Run before building any other skill to discover vault structure
  • After major vault reorganization
  • When onboarding a new vault or switching vaults

Invocation: /jeeves:vault-audit

How It Works (Detailed)

Step 1: Read vault path. Reads the vault location from ~/.claude/jeeves/memory/preferences.md.

Step 2: Walk directory tree. Traverses the top 3 levels of the vault, counting files by type (MD, PDF, DOCX, etc.) per folder. Builds a structural map of the vault.

Step 3: Sample files per folder. For each major folder, reads 3-5 MD files and inspects:

  • Frontmatter fields and structure
  • Heading hierarchy and document structure
  • Wiki-link and cross-reference patterns
  • Naming conventions

Step 4: Deep dive into Meetings. Samples transcription tool output format: notes MD + linked transcript files. Reports frontmatter fields, linking patterns between notes and transcripts, naming conventions.

Step 5: Deep dive into Stakeholders. Samples stakeholder files, reports their structure, fields, and how they link to meetings and projects.

Step 6: Produce structured report. Outputs actual folder names, existing conventions, file type distributions, cross-linking state, and any gaps or inconsistencies discovered.

New Skill: process-meeting

Purpose & Trigger

What it does: This is Karpathy's "Ingest" operation applied to meetings. Takes a meeting (notes MD + transcript) and produces three outputs, touching 10-15 files in the process. This is the core value skill — it transforms raw meeting artifacts into structured, cross-linked knowledge.

When to use:

  • After each meeting: process the notes and transcript
  • Batch: process a backlog of unprocessed meetings
  • Manually: /jeeves:process-meeting

Invocation: /jeeves:process-meeting

How It Works (Detailed)

Output 1: Enhanced meeting note.

  • Structured frontmatter: attendees, projects, meeting_type
  • Wiki-links to stakeholders, projects, and transcript file
  • Related meetings discovery via semantic search + structured overlap + entity co-occurrence
  • ## Related Meetings section with chronological context

Output 2: Tasks and action items.

  • Delegates to task-creator skill
  • Each task linked to: source meeting, project, assignee (stakeholder)

Output 3: Knowledge entries.

  • Appends to _knowledge/log.md with timestamped entry
  • Updates or creates entity pages in _knowledge/entities/
  • Updates _knowledge/index.md with new references
  • Related meetings discovery adds chronological discussion timelines to entity pages

New Skill: task-creator

Purpose & Trigger

What it does: Companion skill to process-meeting. Extracts action items from meeting notes and creates properly linked task entries. Can also be invoked standalone when you want to create tasks from any source.

When to use:

  • Called automatically by process-meeting
  • Standalone: create tasks from any note or ad hoc

Invocation: /jeeves:task-creator

How It Works (Detailed)

Step 1: Extract action items. Identifies who, what, and when from the source material. Parses natural language commitments ("I'll send the report by Friday") into structured tasks.

Step 2: Create task entries. Uses the task format discovered by vault-audit. Respects existing conventions for task files, frontmatter fields, and folder locations.

Step 3: Link tasks. Each task is linked to:

  • Source meeting: Wiki-link back to the meeting note where the action item originated
  • Project: Wiki-link to the relevant project page
  • Assignee: Wiki-link to the stakeholder page for the person responsible

Step 4: Update related files. Adds task references to the source meeting note and relevant project/stakeholder pages.

Updates to Existing Skills

obsidian-knowledge

Current: 2-tier search (Tier 1: smart-search MCP, Tier 2: Grep/Glob fallback).

Update: Add Tier 0 — read _knowledge/index.md FIRST on every query. Then check entity/concept pages in _knowledge/entities/ by direct file read. Then fall back to smart-search (Tier 1) and Grep/Glob (Tier 2). Answers that reveal new knowledge can become new wiki pages.

Why: The index provides a pre-built map of all known entities and their locations. Reading it first avoids expensive search calls for questions that can be answered by direct file reads. Entity pages contain pre-synthesized, cross-referenced information that is higher quality than raw search results.

meeting-prep

Current: Pulls from vault via smart-search + web.

Update: Add structured frontmatter queries for attendees and projects. Pull from _knowledge/entities/ for stakeholder context — their role, past interactions, discussion history. Use entity pages to build attendee profiles with chronological meeting history and key topics discussed.

Why: Meeting prep is dramatically better when it draws from accumulated knowledge about people and topics rather than searching raw notes. Stakeholder entity pages provide a pre-built dossier for each attendee.

session-notes

Current: Writes a session summary MD to the vault.

Update: After writing the session note, trigger knowledge ingestion to update _knowledge/ entity pages. This ensures that knowledge generated during a Claude Code session (research findings, decisions, analysis) is captured in entity pages before the session context is lost.

Why: Without this, session knowledge evaporates. With it, every session contributes to the permanent knowledge base. Entities discovered during research, decisions made, and connections identified all flow into entity pages automatically.

How Everything Cross-Functions

The power of this system comes from the cross-cutting interactions between skills. No single skill does everything — they compose. vault-audit discovers structure, process-meeting ingests content, task-creator extracts actions, and the _knowledge/ layer ties it all together for queries and prep.

InteractionHow It WorksWhy It Matters
vault-audit → process-meeting vault-audit discovers the vault's folder structure, naming conventions, frontmatter patterns, and meeting/transcript format. process-meeting uses these conventions when enhancing meeting notes. process-meeting respects your existing vault conventions instead of imposing its own format.
vault-audit → task-creator vault-audit discovers the existing task format, folder locations, and frontmatter fields. task-creator uses these when creating new tasks. New tasks are consistent with your existing task system.
process-meeting → task-creator → tasks linked process-meeting identifies action items in the meeting, delegates to task-creator which creates linked task entries pointing to the meeting, project, and assignee. Action items are never lost. Every task traces back to its source meeting and forward to its assignee.
process-meeting → _knowledge/ populated Each processed meeting updates _knowledge/log.md, creates or updates entity pages in _knowledge/entities/, and updates _knowledge/index.md. The knowledge graph grows with every meeting. Entity pages accumulate chronological discussion timelines.
obsidian-knowledge → _knowledge/index.md → entity pages On every query, obsidian-knowledge reads _knowledge/index.md first (Tier 0), then relevant entity pages, then falls back to smart-search and Grep/Glob. Queries are faster and higher quality because they tap pre-synthesized knowledge before resorting to search.
meeting-prep → _knowledge/entities/ → attendee profiles When preparing for a meeting, meeting-prep queries frontmatter for attendees and projects, then reads stakeholder entity pages for context, past interactions, and discussion history. Meeting prep draws from accumulated knowledge about people and topics, not just raw note search.
session-notes → _knowledge/ ingestion When a session ends, session-notes writes a summary, then triggers knowledge ingestion to update entity pages with session findings. Research sessions, decisions, and analysis don't evaporate when the session ends.
PARA structure → QMD contexts → search results If using QMD, PARA folder contexts are attached to collections. Search results include this context. Jeeves agents know whether a result comes from an active project or archived reference material.

Implementation Priority

Start here, in this order:

1. vault-audit (Foundation — discover vault structure)

Build this first. It's purely read-only and gives you a complete structural report of your vault: folder names, file type distributions, frontmatter conventions, meeting formats, stakeholder structures, and cross-linking patterns. Everything else depends on this knowledge.

↓

2. process-meeting (Core value — Karpathy's ingest)

The highest-value skill. Takes a meeting (notes + transcript) and produces enhanced notes, linked tasks, and knowledge entries. This is where the Karpathy pattern truly comes alive — each meeting enriches the knowledge graph, entity pages grow richer, and related meetings are discovered automatically.

↓

3. task-creator (Companion — extract action items)

Companion to process-meeting. Creates properly linked task entries from action items. Also usable standalone. Respects the task format discovered by vault-audit.

↓

4. obsidian-knowledge update (Tier 0 — index.md first)

Once _knowledge/ has meaningful content from process-meeting, update obsidian-knowledge to read _knowledge/index.md first on every query. This is when Jeeves queries get dramatically faster and higher quality.

↓

5. meeting-prep update + session-notes update

Update meeting-prep to pull from _knowledge/entities/ for attendee profiles and structured frontmatter queries. Update session-notes to trigger knowledge ingestion after writing session summaries.
