Jeeves Skills for the Knowledge Base

Five new skills form the compilation layer. Three existing skills get updates. Together they implement the full Karpathy LLM Wiki pattern on your PARA vault.

How the Skills Relate to Each Other

# The compilation pipeline: skills trigger each other in sequence wiki-crawl — "What files need attention?" ├── Scans vault for uncovered raw files (no user note, no companion MD) ├── Identifies stale companions (source file changed since last compile) ├── Reports coverage statistics └── Triggers ↓ for uncovered files │ wiki-compile — "Generate companion MDs for uncovered files" ├── Converts raw file to text (MarkItDown for binary formats) ├── LLM generates structured summary with frontmatter ├── Writes companion MD to _wiki/ subfolder └── Triggers ↓ for the new companion │ wiki-ingest — "Extract entities and concepts, build knowledge graph" ├── Reads any source note (user-authored OR auto-generated) ├── Extracts entities (products, people, regulations, companies, studies) ├── Extracts concepts (regulatory pathways, clinical evidence, market dynamics) ├── Creates or updates pages in _knowledge/entities/ and _knowledge/concepts/ ├── Updates _knowledge/index.md and log.md └── Cross-references are bidirectional (source ↔ entity page) # Query and maintenance: used on-demand wiki-query — "Ask questions against the knowledge graph" ├── Tier 0: Read _knowledge/ entity and concept pages directly ├── Tier 1: Semantic search via smart-search/QMD across all MDs ├── Tier 2: Grep/Glob fallback └── Synthesize answer with source attribution wiki-lint — "Health check the knowledge base" ├── Orphaned companions, stale content, uncovered files ├── Missing cross-references, index completeness ├── Contradiction detection across sources └── Produces actionable report

New Skill: wiki-crawl

Purpose & Trigger

What it does: Walks your entire vault (Projects/, Areas/, Resources/ — not Archives) and produces a complete inventory of all raw files and their coverage status. For each non-Markdown file, it determines whether you've written a note for it, whether an auto-generated companion exists, or whether it's completely uncovered.

When to use:

  • First time: Run on your entire vault to understand current coverage
  • Periodically: Run weekly or after adding many files
  • Automatically: File watcher can trigger for individual new files

Invocation: /jeeves:wiki-crawl or /jeeves:wiki-crawl ~/vault/Resources/Literature

How It Works (Detailed)

Step 1: Walk the vault. Recursively scan all folders under Projects/, Areas/, Resources/. Skip folders starting with _ (like _wiki/, _knowledge/) and skip Archives/ entirely. Collect all non-MD files (PDF, DOCX, PPTX, XLSX, PNG, JPEG, etc.).

Step 2: For each raw file, check coverage. Look for an associated Markdown note by checking:

  • Wiki-links: Any MD in the same folder or parent folder contains [[filename]]
  • Markdown links: [text](./filename) or [text](filename)
  • Frontmatter: source: filename in any nearby MD
  • Name matching: filename.md exists next to filename.pdf
  • Companion: _wiki/filename.md exists

Step 3: Produce report. Categorize each file as: covered (has user note), auto-covered (has companion MD), stale (companion older than source), or uncovered (no note at all).

New Skill: wiki-compile

Purpose & Trigger

What it does: For a raw file that has no user note, generates a structured Markdown companion file. This is the "gap-filler" — it ensures every file in your vault is represented as searchable, LLM-readable Markdown text.

When to use:

  • Called by wiki-crawl for uncovered files
  • Triggered by file watcher when a new raw file appears and no note is found
  • Manually: /jeeves:wiki-compile path/to/file.pdf

Does NOT: Overwrite any existing user notes. Only creates new files in _wiki/ subfolders. If you later write your own note for a file, the companion becomes redundant (wiki-lint will flag it).

How It Works (Detailed)

Step 1: Convert file to text. Uses MarkItDown (the same library in smart-search) to convert binary formats to Markdown. Handles: PDF (text extraction + table recognition), DOCX (paragraphs + tables + headers), PPTX (slide-by-slide with speaker notes), XLSX (sheet-by-sheet with headers), images (vision model generates description), HTML (article text extraction).

Step 2: LLM generates summary. The converted text is sent to Claude Sonnet (cost-efficient) with a structured prompt requesting:

  • YAML frontmatter: source_path, source_hash (SHA-256), source_mtime, file_type, auto_generated: true, tags (3-7 descriptive tags), created_at, updated_at
  • One-paragraph summary capturing the document's purpose and key content
  • Structured key takeaways as bullet points (5-15 depending on document size)
  • Identified entities and concepts (used by wiki-ingest in the next step)
  • Obsidian wiki-links to any existing _knowledge/ pages that are relevant
  • A source attribution link back to the original file

Step 3: Write to _wiki/. Creates the _wiki/ subfolder if it doesn't exist. Writes the companion MD. The filename matches the source file (e.g., roadmap.xlsx → _wiki/roadmap.md).

New Skill: wiki-ingest

Purpose & Trigger

What it does: This is the most important skill. It reads a source note (either your handwritten note or an auto-generated companion) and extracts entities and concepts into the cross-cutting _knowledge/ graph. This is what makes the knowledge base compound over time — each note contributes to entity and concept pages that grow richer with every ingestion.

When to use:

  • After wiki-compile generates a new companion MD
  • After you write or update a note
  • After session-notes creates a session summary
  • Manually: /jeeves:wiki-ingest path/to/note.md
  • Batch: /jeeves:wiki-ingest path/to/folder/ (processes all MDs in folder)

Model: Claude Opus (cross-referencing and entity extraction require deep reasoning about what constitutes a meaningful entity vs noise).

How It Works (Detailed)

Step 1: Read the source note. Parse the MD file, extract frontmatter metadata, understand the document type and context (meeting note? literature note? project plan?).

Step 2: Extract entities. The LLM identifies specific, named things mentioned:

  • Products: CAMS, EndoVision, AI Colonoscopy v2
  • People: Dr. Smith, Prof. Mueller, stakeholder names
  • Companies: Acme AI, MedTech Corp, competitors
  • Regulations: FDA 510(k), EU MDR, IEC 62304, ISO 13485
  • Studies: CLEOPATRA-III Trial, Smith 2025 RCT
  • Standards & Frameworks: GRADE, QUADAS-2, IEC 62366

Step 3: Extract concepts. Abstract topics that span multiple entities: regulatory strategy, clinical evidence planning, market positioning, software lifecycle.

Step 4: Update knowledge pages. For each entity/concept:

  • If page exists in _knowledge/: Read the current page, append new information under a source-attributed section. Increment source_count in frontmatter. Update last_updated timestamp.
  • If page is new: Create it using the template from _knowledge/SCHEMA.md. Populate with initial content from this source.
  • Add bidirectional wiki-links: source note links to entity page, entity page links to source note.

Step 5: Update index.md and log.md. Add/update entries in the master catalog. Append chronological record.

New Skill: wiki-query

Purpose

What it does: Answers questions using the knowledge base. Implements the three-tier search pattern (Tier 0: direct knowledge page reads, Tier 1: semantic search, Tier 2: Grep/Glob fallback). This skill is also used internally by other Jeeves agents and skills when they need vault knowledge.

Invocation: /jeeves:wiki-query "What regulatory pathway should we use for CAMS in the EU?"

How it differs from smart-search skill: The existing smart-search skill is a direct proxy to the search engine. wiki-query is smarter — it checks _knowledge/ entity pages first (often answering without any search), falls back to search, and synthesizes answers with full source attribution. It understands the knowledge graph structure.

Optional behavior: If the query reveals new information or connections not yet in the knowledge graph, wiki-query can optionally trigger wiki-ingest to update entity/concept pages with its findings. This means the act of querying can itself improve the knowledge base.

New Skill: wiki-lint

Purpose

What it does: Periodic health check for the knowledge base. Scans for problems that naturally accumulate over time: files deleted but companions still exist, sources updated but knowledge pages not refreshed, contradictions between sources, index entries missing. Produces a human-readable report with actionable items.

Invocation: /jeeves:wiki-lint (run weekly or on-demand)

Checks performed: Orphaned companions (source deleted), stale companions (source newer), uncovered files, redundant companions (you wrote a note after auto-generation), missing cross-references, stale entity pages, contradictions, index completeness. See the Workflow page for a detailed example of lint output.

Updates to Existing Skills

obsidian-knowledge

Current: 2-tier search (Tier 1: smart-search MCP, Tier 2: Grep/Glob fallback).

Update: Add Tier 0 before current tiers. Tier 0 checks _knowledge/ entity and concept pages by direct file read. This is the fastest and highest-quality search path because knowledge pages contain pre-synthesized, cross-referenced information. The existing Tier 1 and Tier 2 remain as fallbacks.

Why: Entity pages often answer questions directly without needing semantic search. A question about "FDA 510(k)" can be answered by reading _knowledge/entities/fda-510k.md instead of searching across hundreds of document chunks.

session-notes

Current: Writes a session summary MD to the vault.

Update: After writing the session note, automatically trigger wiki-ingest on the new note. This ensures that knowledge generated during a Claude Code session (research findings, decisions, analysis) is captured in the knowledge graph before the session context is lost.

Why: Without this, session knowledge evaporates. With it, every session contributes to the permanent knowledge base. Entities discovered during research, decisions made, and connections identified all flow into entity/concept pages automatically.

daily-briefing

Current: Pulls from vault via smart-search + web.

Update: Include a "Knowledge Base Updates" section showing: new entity/concept pages created since last briefing, updated pages (what changed), any contradictions flagged by wiki-lint, and uncovered files count. This gives you a daily pulse on how the knowledge base is growing.

Why: Keeps you aware of knowledge base health without needing to run wiki-lint manually. Surfaces interesting connections that were discovered automatically.

How Everything Cross-Functions

The power of this system comes from the cross-cutting interactions between tools. No single tool does everything — they compose. Smart-search/QMD handles search. Jeeves skills handle compilation and knowledge management. The PARA vault provides organizational structure. Obsidian provides the viewing and editing interface. Here's how they interact:

InteractionHow It WorksWhy It Matters
File watcher → wiki-crawl → wiki-compile Smart-search's Watchdog detects a new file. If it's a non-MD file, it checks coverage. If uncovered, wiki-compile generates a companion MD. Automatic: new files get Markdown representation without manual effort.
wiki-compile → wiki-ingest → _knowledge/ After a companion MD is generated, wiki-ingest extracts entities and concepts into the knowledge graph. The knowledge graph grows automatically with every new file.
session-notes → wiki-ingest → _knowledge/ When you end a session, session-notes writes a summary, then wiki-ingest captures the session's knowledge. Research sessions, decisions, and analysis don't evaporate when the session ends.
Jeeves agents → wiki-query → _knowledge/ + search When regulatory-sme, clinical-sme, or any agent needs vault knowledge, it goes through wiki-query's 3-tier search (knowledge graph first, then semantic search, then Grep). Agent answers are richer and faster because they tap pre-synthesized knowledge.
meeting-prep → _knowledge/entities/ → attendee pages When preparing for a meeting, Jeeves reads entity pages for attendees (their background, past interactions) and topic pages (what we know about the discussion topics). Meeting prep is dramatically better because it draws from accumulated knowledge about people and topics.
wiki-lint → daily-briefing Lint findings (contradictions, stale content, uncovered files) are surfaced in the daily briefing. You stay aware of knowledge base health without running manual audits.
PARA structure → QMD contexts → search results If using QMD, PARA folder contexts are attached to collections ("these are active project resources", "these are reference materials"). Search results include this context. Jeeves agents know whether a result comes from an active project or archived reference material, improving decision-making.
_knowledge/SCHEMA.md → all wiki skills The schema file defines templates, conventions, and rules that all wiki skills follow when creating or updating content. Consistency across the entire knowledge base. Every entity page looks the same, every cross-reference follows the same format.

Implementation Priority

Start here, in this order:

1. wiki-crawl (Foundation)

Build this first. It's purely read-only and gives you immediate visibility into your vault's coverage. You'll see exactly how many files have notes, how many don't, and where the biggest gaps are. This informs everything else.

↓

2. wiki-compile (Gap-filling)

Build this second. It fills the gaps that wiki-crawl identified. After running this on your vault, every file has a Markdown representation. The vault is now fully searchable as text.

↓

3. wiki-ingest (The Compounding Layer)

This is the highest-value skill. It builds the knowledge graph from all your notes. Start with a small subset of your vault (e.g., one project folder), validate the entity extraction quality, then scale to the full vault. This is where Karpathy's pattern truly comes alive.

↓

4. wiki-query + obsidian-knowledge update

Once the knowledge graph has meaningful content, build the query layer. Update obsidian-knowledge to include Tier 0 (knowledge graph direct reads). This is when Jeeves agents get dramatically better.

↓

5. wiki-lint + daily-briefing update

Maintenance and monitoring. Build once the system has been running for a while and needs health checks.
