System Workflow

Karpathy's three operations — Ingest, Query, Lint — applied to the Meetings pilot. Each workflow visualized step-by-step.

Workflow 1: Ingest (process-meeting skill)

A meeting concludes and you have two artifacts: a meeting notes Markdown file and a transcript. The process-meeting skill fires and touches 10–15 files across the vault.

Step 1: Read Meeting Notes & Transcript

The skill reads the meeting notes MD file and its companion transcript. These are the raw inputs. Everything downstream is derived from these two artifacts.

↓

Step 2: Enhance with Structured Frontmatter

The skill enriches the meeting note with structured YAML frontmatter:

• date: — meeting date
• attendees: — list of participants
• projects: — linked project identifiers
• meeting_type: — e.g., standup, design-review, regulatory-sync
• transcript: — relative link to the transcript file

This structured metadata powers downstream discovery and filtering.

↓

Step 3: Add Wiki-Links to Stakeholders & Projects

The skill scans the note body and inserts [[wiki-links]] to:

• Stakeholder files — e.g., [[Dr. Muller]], [[Sarah Chen]]
• Project files — e.g., [[CAMS]], [[Regulatory Submission]]

These bidirectional links connect the meeting into the broader knowledge graph.

↓

Step 4: Discover Related Meetings

This is the high-value step. The skill finds related meetings using three complementary strategies:

4a. Semantic search (smart-search): Extract key topics from the meeting and query them against all meeting notes. Surfaces meetings that discuss similar concepts even if they use different terminology.

4b. Structured overlap: Match frontmatter fields — same attendees, same projects, overlapping tags. Fast, precise, and deterministic.

4c. Entity co-occurrence: If two meetings mention the same named entities (stakeholders + topics), they are related. E.g., two meetings both mentioning "Dr. Muller" and "510(k)" are almost certainly connected.

↓

Step 5: Add "Related Meetings" Section

The skill appends a ## Related Meetings section to the note with wiki-links to discovered meetings:

## Related Meetings
- [[2026-03-15 Regulatory Sync]] — also discussed 510(k) timeline
- [[2026-02-28 Strategy Review]] — initial pathway decision
- [[2026-01-10 Competitive Intel]] — Competitor X 510(k) precedent

↓

Step 6: Extract Tasks & Action Items

The skill identifies action items and delegates them to the task-creator skill. Each task gets an owner (from attendees), a due date if mentioned, and a backlink to the source meeting. Tasks flow into the project's task tracking.

↓

Step 7: Extract Knowledge & Update Entity Pages

The skill extracts knowledge entries from the meeting and updates the knowledge base:

• Entity pages in _knowledge/ are updated — new facts appended with source attribution and date
• Source page created in _knowledge/sources/meetings/ documenting what the wiki learned from this meeting — what entities were found, what concepts were discussed, and what pages were updated
• The source page creates the cross-reference graph: source page → entity pages → other source pages mentioning the same entity
• hot.md is updated with the latest context from this meeting
• log.md gets a new chronological entry recording what was processed
• index.md is updated with any new entity or concept pages created

Entity pages accumulate a timeline of discussions, building a living history of each topic.

↓

Step 8: File Touch Summary

Target: 10–15 files touched per meeting. Typical breakdown:
• 1 meeting note (enhanced with frontmatter, wiki-links, related meetings section)
• 2–4 entity pages in _knowledge/ (updated with new facts)
• 1 source page in _knowledge/sources/meetings/ (cross-reference record)
• 1 hot.md (updated with latest context)
• 1–3 stakeholder files (backlink added)
• 1–2 project files (backlink added)
• 1 log.md (new entry appended)
• 1 index.md (updated if new pages created)
• 1–3 related meeting notes (reciprocal links added)
• 0–2 task files (from extracted action items)

Example: Entity Page Timeline

After several meetings, an entity page builds up a rich discussion timeline automatically:

# FDA 510(k) — CAMS ## Discussion Timeline - [[2026-01-10]] — Noted Competitor X got 510(k) in 6 months - [[2026-02-28]] — Decided to pursue 510(k) over De Novo - [[2026-03-15]] — Dr. Muller raised IRB timeline concern - [[2026-04-01]] — Updated: submission target Q3 2026

Why this matters: When someone asks "what's our 510(k) status?" the answer is already compiled on this entity page — no search needed. This is Karpathy's key insight: synthesis is pre-computed at ingest time, not at query time.

Workflow 2: Query (read index.md first)

You ask: "What regulatory pathway should we use for CAMS in the US and EU?" — The system follows Karpathy's pattern: read the index first to know what exists, then drill down.

Step 1: Read _knowledge/index.md FIRST

This is the critical step that makes everything else efficient. Before any search, the agent reads _knowledge/index.md to learn what entity and concept pages exist. This is Karpathy's pattern: the index is a table of contents for the entire knowledge base. The agent now knows:

• Which entities have pages (e.g., fda-510k.md, eu-mdr.md, product-cams.md)
• Which concepts are tracked (e.g., regulatory-pathways-ai-devices.md)
• What categories and topics the vault covers

This is fast, deterministic, and costs almost nothing. No embeddings, no vector search — just reading a file.

↓

Step 2: Read Relevant Entity & Concept Pages

Based on the index, the agent reads the specific pages that match the query:

• _knowledge/entities/fda-510k.md — pre-synthesized 510(k) pathway info from multiple meetings
• _knowledge/entities/eu-mdr.md — EU regulatory pathway information
• _knowledge/entities/product-cams.md — product-specific regulatory context
• _knowledge/concepts/regulatory-pathways-ai-devices.md — broader strategy overview

This is often sufficient to answer the question because entity pages already aggregate knowledge from all sources. The synthesis was pre-computed during ingest (Workflow 1).

↓

Step 3: Semantic Search via smart-search (if needed)

If the entity pages don't fully answer the question (e.g., the question is about a very specific detail not yet captured on an entity page), the agent runs a semantic search:

• Query: "CAMS regulatory pathway US EU 510k MDR"
• Searches across: all meeting notes, user notes, companion MDs, and knowledge pages
• Returns: ranked results with relevance scores, source file paths, and text snippets
• The agent reads the top results and incorporates them into its answer

↓

Step 4: Grep/Glob Fallback (if search unavailable)

If the search engine is down or unreachable, Jeeves falls back to pattern-based file search using Grep and Glob across the vault. This is slower and less precise but ensures the system degrades gracefully.

↓

Step 5: Synthesize Answer & Close the Loop

The agent synthesizes information from all tiers into a cohesive answer. Every claim includes source attribution — links back to specific meeting notes and original documents.

Query → Knowledge loop: If the query reveals a gap — information that should be on an entity page but isn't — the answer itself can become a new wiki page or trigger an update to existing entity pages. The query operation feeds back into the knowledge graph, closing the loop.

Filing valuable answers: Particularly valuable answers can be filed to _knowledge/analysis/ as new wiki pages. These analysis pages compound into the knowledge base, making future queries richer and reducing repeated synthesis work.

Karpathy's hierarchy: index.md → entity/concept pages → smart-search → Grep/Glob. Each tier is cheaper and faster than the next. Most questions are answered by Tier 1-2 alone. The index is the routing table for the entire knowledge base.

Workflow 3: Periodic Health Check (Lint) Future

Run weekly or on-demand via /jeeves:wiki-lint. Ensures the knowledge base stays healthy. Not built in the pilot — this is a planned future capability.

Checks Performed

Orphaned Companions A companion MD exists in _wiki/ but its source file has been deleted or moved. Action: flag for removal or archival.
Stale Companions The source file has been modified since the companion MD was generated (source_hash mismatch). Action: flag for re-compilation.
Uncovered Files Raw files that have neither a user note nor a companion MD. Action: flag for compilation.
Redundant Companions You wrote a note for a file that already has an auto-generated companion. The companion is now redundant. Action: flag companion for removal.
Missing Cross-references An entity is mentioned in a source note but not linked to its entity page. Action: suggest adding the link.
Stale Entity Pages An entity page hasn't been updated in a long time, but related sources have changed. Action: flag for re-ingestion.
Contradictions Two source notes make conflicting claims about the same entity or concept. Action: flag with both sources for human review.
Index Completeness Not all entity/concept pages are listed in _knowledge/index.md. Action: add missing entries.

Workflow 4: End of Session

When you finish a work session with Jeeves, the session-notes skill can trigger knowledge base updates.

You invoke /jeeves:session-notes

Jeeves summarizes what was discussed, decisions made, research findings, and open items into a session note MD file in your vault (existing behavior).

↓

wiki-ingest is triggered automatically

The session note is treated like any other user note. The LLM extracts entities and concepts mentioned during the session, updates existing knowledge pages with new information, and creates new entity/concept pages if needed. Your session's work is now part of the permanent knowledge graph.

↓

Knowledge compounds over time

Each session adds to the knowledge graph. Over weeks and months, entity pages become rich, cross-referenced resources. A page like product-cams.md accumulates information from dozens of meetings, research sessions, and planning activities — automatically. This is the "compounding artifact" Karpathy describes.
