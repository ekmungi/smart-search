System Workflow

What happens step-by-step when files are added, changed, or queried. Three core workflows visualized in detail.

Workflow 1: A New File Arrives

You save a new PDF (e.g., a research paper) into Resources/Literature/. Here's what happens:

Step 1: File Watcher Detects Change

The file system watcher (smart-search's Watchdog or a standalone MCP) monitors your vault folders (Projects/, Areas/, Resources/). It detects the new PDF file. It ignores folders starting with _ (to avoid processing its own output) and ignores the Archives/ folder entirely. The watcher debounces rapid changes (2-second window) to avoid triggering on partial file saves.

↓

Step 2: Coverage Check (wiki-crawl logic)

Before doing anything, the system checks: does this file already have an associated Markdown note? It looks for:
(a) An MD file in the same folder that contains a wiki-link [[filename.pdf]] or markdown link [text](filename.pdf)
(b) An MD file with matching name (e.g., paper-name.md next to paper-name.pdf)
(c) An MD file with source: filename.pdf in its frontmatter

If a user note is found: Skip companion MD generation (your note IS the companion). Go directly to Step 4 (entity extraction).
If no user note found: Proceed to Step 3 (auto-generate companion).

↓

Step 3: Companion MD Generation (wiki-compile skill)

Since no user note exists, the system generates one automatically:

3a. File Conversion: The raw PDF is converted to plain text using MarkItDown (the same library smart-search uses today). This handles tables, headers, metadata extraction. For images (PNG/JPEG), a vision model generates a text description.

3b. LLM Summarization: The converted text is sent to an LLM (Sonnet, for cost efficiency) with a structured prompt that produces: YAML frontmatter (source_path, source_hash, file_type, tags), a one-paragraph summary, key takeaways as bullet points, identified entities and concepts, and links to relevant _knowledge/ pages if they exist.

3c. Write to _wiki/: The companion MD is saved to Resources/Literature/_wiki/paper-name.md with auto_generated: true in frontmatter. The _wiki/ subfolder is created if it doesn't exist.

↓

Step 4: Entity & Concept Extraction (wiki-ingest skill)

The LLM reads the source note (either your user note from Step 2, or the auto-generated companion from Step 3) and identifies:

Entities: Specific things mentioned — products (CAMS), people (Dr. Smith), companies (Acme AI), regulations (FDA 510(k)), studies (CLEOPATRA-III), devices, standards.

Concepts: Abstract topics discussed — regulatory pathways, clinical evidence requirements, AI safety in medical devices, software lifecycle management.

For each entity/concept found:
• Check if a page already exists in _knowledge/entities/ or _knowledge/concepts/
• If exists: append new information with source attribution (never replace, always accumulate)
• If new: create a new page using the template from _knowledge/SCHEMA.md
• Add bidirectional cross-references (the source note links to the entity page, the entity page links back to the source)

↓

Step 5: Index & Log Update

index.md: The master catalog is updated with entries for any new entity or concept pages created, and updated entries for modified pages. Each entry has the page title, a one-line summary, category, and last-updated date.

log.md: A new line is appended to the chronological log: ## [2026-04-06] ingest | smith-2025-rct.pdf | Created entities: product-cams, fda-510k | Updated concepts: clinical-evidence-requirements

This provides a full audit trail of what was processed, when, and what knowledge pages were affected.

↓

Step 6: Search Index Auto-Update

The search engine (smart-search or QMD) detects the new MD files that were created or modified in Steps 3-5. It automatically indexes them:
• The companion MD (or your user note) gets chunked and embedded for semantic search
• The updated entity/concept pages get re-indexed
• The raw PDF itself is not indexed — only its MD representation

Search results now include the new knowledge. The next time you or a Jeeves agent searches for related topics, this content will surface.

Workflow 2: Asking a Question

You ask: "What regulatory pathway should we use for CAMS in the US and EU?"

Step 1: Jeeves Routes to Agent

Jeeves analyzes the question and routes it to the regulatory-sme agent (or the wiki-query skill for direct wiki search). The agent is specialized in FDA, EU MDR, and global regulatory pathways.

↓

Step 2: Tier 0 — Check Knowledge Graph First

Before doing any search, the agent checks _knowledge/ directly:

• Reads _knowledge/entities/fda-510k.md — contains pre-synthesized 510(k) pathway information from multiple meetings, guidance documents, and project plans
• Reads _knowledge/entities/eu-mdr.md — EU regulatory pathway information
• Reads _knowledge/entities/product-cams.md — product-specific regulatory context
• Reads _knowledge/concepts/regulatory-pathways-ai-devices.md — broader regulatory strategy overview

This is often sufficient to answer the question because entity pages already aggregate knowledge from all sources. No search engine needed. This is the key advantage of the Karpathy pattern: synthesis is pre-computed.

↓

Step 3: Tier 1 — Semantic Search (if needed)

If the entity pages don't fully answer the question (e.g., the question is about a very specific detail), the agent runs a semantic search via smart-search/QMD MCP:

• Query: "CAMS regulatory pathway US EU 510k MDR"
• Searches across: all user notes + companion MDs in _wiki/ + knowledge pages in _knowledge/
• Returns: ranked results with relevance scores, source file paths, and text snippets
• The agent reads the top results and incorporates them into its answer

↓

Step 4: Tier 2 — Grep/Glob Fallback (if search unavailable)

If the search engine is down or unreachable, Jeeves falls back to pattern-based file search using Grep and Glob across the vault. This is slower and less precise but ensures the system degrades gracefully.

↓

Step 5: Synthesize Answer

The agent synthesizes information from all tiers into a cohesive answer. Every claim includes source attribution — links back to specific notes, companion MDs, and ultimately the original raw documents. If the query reveals new knowledge worth preserving, the agent can optionally trigger wiki-ingest to update the knowledge graph with its findings.

Workflow 3: Periodic Health Check (Lint)

Run weekly or on-demand via /jeeves:wiki-lint. Ensures the knowledge base stays healthy.

Checks Performed

Orphaned Companions A companion MD exists in _wiki/ but its source file has been deleted or moved. Action: flag for removal or archival.
Stale Companions The source file has been modified since the companion MD was generated (source_hash mismatch). Action: flag for re-compilation.
Uncovered Files Raw files that have neither a user note nor a companion MD. Action: flag for compilation.
Redundant Companions You wrote a note for a file that already has an auto-generated companion. The companion is now redundant. Action: flag companion for removal.
Missing Cross-references An entity is mentioned in a source note but not linked to its entity page. Action: suggest adding the link.
Stale Entity Pages An entity page hasn't been updated in a long time, but related sources have changed. Action: flag for re-ingestion.
Contradictions Two source notes make conflicting claims about the same entity or concept. Action: flag with both sources for human review.
Index Completeness Not all entity/concept pages are listed in _knowledge/index.md. Action: add missing entries.

Workflow 4: End of Session

When you finish a work session with Jeeves, the session-notes skill can trigger knowledge base updates.

You invoke /jeeves:session-notes

Jeeves summarizes what was discussed, decisions made, research findings, and open items into a session note MD file in your vault (existing behavior).

↓

wiki-ingest is triggered automatically

The session note is treated like any other user note. The LLM extracts entities and concepts mentioned during the session, updates existing knowledge pages with new information, and creates new entity/concept pages if needed. Your session's work is now part of the permanent knowledge graph.

↓

Knowledge compounds over time

Each session adds to the knowledge graph. Over weeks and months, entity pages become rich, cross-referenced resources. A page like product-cams.md accumulates information from dozens of meetings, research sessions, and planning activities — automatically. This is the "compounding artifact" Karpathy describes.
