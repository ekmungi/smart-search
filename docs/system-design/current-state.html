Current State of the System

Where smart-search (v0.15.1) and Jeeves (v1.11.1) stand today, what's missing, and the immediate next step.

Smart-search — v0.15.1 Production Ready

What it is: A local-first, offline semantic search engine. Indexes documents in the Obsidian vault and provides hybrid search (keyword + vector + reranking). Runs as HTTP server (port 9742) with MCP server, REST API, CLI, and Tauri desktop app. RAM: ~130MB (ONNX embeddings + TinyBERT reranker). No GPU required.

What's Working Well

File Format Support14 formats via MarkItDown: PDF, DOCX, XLSX, PPTX, EPUB, CSV, JSON, HTML, MD, TXT, MSG, IPYNB. Converts everything to Markdown before indexing.
Hybrid SearchBM25 full-text + vector semantic (Snowflake Arctic Embed M v2.0, 256-dim), combined via RRF (k=60). Cross-encoder reranking (TinyBERT, ~15MB). MMR for diversity.
MCP Integration11 MCP tools: knowledge_search, knowledge_ingest, find_related, read_note, knowledge_stats, folder management, temp indexes.
File WatchingWatchdog-based auto-detection with 2-second debounce. Per-folder cancellation. Thread-safe add/remove at runtime.
Desktop AppTauri v2 with system tray, Ctrl+Space quick search, folder management UI, indexing log.
PerformanceVector search <50ms, BM25 <30ms, reranking 30-60ms. Idle RAM ~130MB. ONNX with GPU auto-detection (CUDA/DirectML).
Test Coverage425+ pytest tests. Comprehensive coverage of indexing, search, chunking, and API.

Current Limitations

No Knowledge CompilationIndexes raw documents but does not synthesize, cross-link, or build entity pages. Search engine only, not knowledge manager. Key Gap
No Query ExpansionUnlike QMD, does not use LLM-powered query variants to improve recall. Searches only the literal query. Future: port from QMD
No Context SystemDoesn't attach metadata about what a folder represents. PARA context is lost during search. Future: port from QMD
No OCRImage-based PDFs and scanned documents can't be indexed. Flagged as needs_ocr for future work.
Maintenance BurdenComplex Python codebase (FastAPI, LanceDB, ONNX, Tauri, Watchdog, 425+ tests). Significant for a non-developer to maintain.

Smart-search Architecture

# How smart-search currently processes files: Raw File (e.g., report.pdf) ↓ MarkItDown Conversion — Converts PDF/DOCX/PPTX/XLSX to Markdown text ↓ Markdown Chunker — Splits on heading boundaries (# / ## / ###), ~200 words per chunk ↓ ONNX Embedder — Snowflake Arctic Embed M v2.0, 256-dim Matryoshka vectors ↓ LanceDB (vector storage) + SQLite FTS5 (keyword index) ↓ # At query time: Query → Vector Search + BM25 → RRF Fusion → Cross-encoder Rerank → MMR Diversity → Results

In the new architecture: Smart-search continues indexing MD files as today. The new process-meeting skill handles the knowledge compilation layer (entity extraction, cross-linking, knowledge graph). Smart-search provides the search backbone; Jeeves skills provide the intelligence. Three QMD innovations (query expansion, path context, position-aware blending) are tracked as future smart-search improvements.

Jeeves — v1.11.1 Production Ready

What it is: A Claude Code plugin with 8 specialized AI agents and 22 domain skills for a Global AI Product Manager in healthcare/medical devices. Jeeves orchestrates research, analysis, and document creation. Smart-search is one of its tools for finding information in the vault.

What's Working Well

22 Domain Skillsmeeting-prep, partnership-evaluation, market-research, technical-research, daily-briefing, project-docs, prioritization, product-requirements, roadmap-planning, session-notes, and more.
8 Specialized Agentsregulatory-sme, clinical-sme, research-analyst, technical-researcher, partnership-evaluator, user-researcher, product-manager, strategist.
meeting-prep Skill4-layer attendee research, 6 meeting type frameworks, vault-first context search, stakeholder power mapping, pre-wiring strategy.
Memory SystemCross-session persistent memory: preferences.md, learnings.md (curated), learnings-staging.md (pending). Human-in-the-loop curation.
obsidian-knowledge2-tier vault search: Tier 1 semantic (smart-search MCP), Tier 2 keyword (Grep/Glob). PARA-aware search patterns.
54 Structural TestsPre-commit validation of manifest, agents, skills, persona, rules, and cross-references.

Current Limitations

No Knowledge CompilationNo skills for entity extraction, cross-linking, or building knowledge pages. Can search but can't synthesize. Key Gap
No Vault Structure AwarenessDoesn't understand PARA structure. Can't distinguish "this is a Project" from "this is a Resource." Key Gap
No Meeting Processingmeeting-prep prepares FOR meetings. No skill to process AFTER meetings (extract entities, link stakeholders, create tasks). Key Gap
Stakeholders Not LinkedStakeholder MD files exist in Resources but aren't linked to meeting notes. No bidirectional wiki-links. Key Gap
No Task ExtractionAction items from meetings aren't automatically extracted and linked to projects/assignees.
No index.md / log.mdNo Karpathy-style master catalog or chronological record. Each query re-discovers knowledge from scratch.

How They Connect Today

# Current flow (no knowledge layer): You ask: "Prepare me for the FDA meeting tomorrow" ↓ Jeeves routes to meeting-prep skill ↓ meeting-prep searches vault: Tier 1: smart-search MCP → knowledge_search("FDA meeting attendees") Returns: raw document chunks with file paths and scores Tier 2: Grep/Glob fallback ↓ Jeeves synthesizes meeting brief from raw chunks # The problem: # - Returns raw document chunks, not synthesized knowledge # - No entity pages for attendees (must re-search every time) # - Stakeholder files exist but aren't linked to meetings # - No discussion timeline showing how topics evolved across meetings # - No index.md to quickly locate what's known

What Changes with the Knowledge Base

# Future flow (with _knowledge/ layer): You ask: "Prepare me for the FDA meeting tomorrow" ↓ Jeeves routes to meeting-prep skill ↓ meeting-prep searches wiki-first: Tier 0: Read _knowledge/index.md — knows what entities/concepts exist → _knowledge/entities/fda-510k.md — pre-synthesized regulatory pathway → _knowledge/entities/dr-muller.md — attendee profile + meeting history → Discussion timeline: all meetings where FDA was discussed, chronologically Tier 1: smart-search semantic search (if Tier 0 insufficient) Tier 2: Grep/Glob fallback ↓ Jeeves produces a much richer brief: - Entity pages provide pre-synthesized context - Discussion timelines show how topics evolved - Related meetings surface connections you'd miss - Stakeholder profiles include full meeting history
