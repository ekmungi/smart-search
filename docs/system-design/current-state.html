Current State of the System

Where smart-search (v0.15.1) and Jeeves (v1.11.1) stand today, and what's missing for the knowledge base vision.

Smart-search — v0.15.1 Production Ready

What it is: A local-first, offline semantic search engine that indexes documents in your Obsidian vault and provides hybrid search (keyword + vector + reranking). It runs as an HTTP server on your machine with multiple client interfaces: MCP server for Claude Code, REST API, CLI, and a Tauri desktop app.

What's Working Well

File Format Support14 formats via MarkItDown: PDF, DOCX, XLSX, PPTX, EPUB, CSV, JSON, HTML, MD, TXT, MSG, IPYNB, and more. Converts everything to Markdown before indexing.
Hybrid SearchBM25 full-text search + vector semantic search, combined via Reciprocal Rank Fusion (RRF, k=60). Cross-encoder reranking (TinyBERT) on top-20 results. Maximum Marginal Relevance (MMR) for diversity.
MCP Integration11 MCP tools available to Claude Code/Jeeves: knowledge_search, knowledge_ingest, find_related, read_note, knowledge_stats, folder management, temp indexes.
File WatchingWatchdog-based automatic detection of new/changed files. Debounced re-indexing with per-folder cancellation.
Desktop AppTauri v2 desktop app with system tray, Ctrl+Space quick search, folder management UI, indexing log.
PerformanceVector search <50ms, BM25 <30ms, reranking 30-60ms. Idle RAM <200MB. ONNX embeddings with GPU auto-detection.
Test Coverage425+ pytest tests with slow test markers. Comprehensive coverage of indexing, search, chunking, and API.

Current Limitations

No Knowledge CompilationSmart-search indexes raw documents but does not synthesize, cross-link, or build entity pages. It's a search engine, not a knowledge manager. Key Gap
No Query ExpansionUnlike QMD, smart-search does not use LLM-powered query expansion to improve search recall. Searches only the literal query (after normalization).
Indexes Raw Files DirectlyCurrently embeds chunks from converted binary files. In the new architecture, it should only need to index MD files (companion MDs + knowledge pages).
No Context SystemUnlike QMD's collection context, smart-search doesn't attach metadata about what a folder represents. PARA context is lost during search.
No OCRImage-based PDFs and scanned documents can't be indexed yet. Flagged in SQLite as needs_ocr for future work.
Maintenance BurdenYou (a scientist, not a developer) maintain a complex Python codebase with FastAPI, LanceDB, ONNX, Tauri, Watchdog, and 425+ tests. This is significant overhead.

Smart-search Architecture (Current)

# How smart-search currently processes files: Raw File (e.g., report.pdf) ↓ MarkItDown Conversion — Converts PDF/DOCX/PPTX/XLSX to Markdown text ↓ Markdown Chunker — Splits on heading boundaries (# / ## / ###), ~200 words per chunk ↓ ONNX Embedder — Snowflake Arctic Embed M v2.0, 256-dim Matryoshka vectors ↓ LanceDB (vector storage) + SQLite FTS5 (keyword index) ↓ # At query time: Query → Vector Search + BM25 → RRF Fusion → Cross-encoder Rerank → MMR Diversity → Results

What changes in the new architecture: Smart-search's MarkItDown conversion capability moves to the compilation layer (Jeeves skill). Smart-search itself only indexes pre-processed MD files. This dramatically simplifies its role and reduces the maintenance surface. Alternatively, QMD could replace it entirely (see comparison).

Jeeves — v1.11.1 Production Ready

What it is: A Claude Code plugin providing 8 specialized AI agents and 22 domain skills for a Global AI Product Manager working in healthcare/medical devices. Jeeves is the "brain" that orchestrates research, analysis, and document creation. Smart-search is one of its tools for finding information in the vault.

What's Working Well

8 Specialized Agentsregulatory-sme, clinical-sme, research-analyst, technical-researcher, partnership-evaluator, user-researcher, product-manager, strategist. Each with deep domain methodology.
22 Domain Skillsmeeting-prep, partnership-evaluation, market-research, technical-research, daily-briefing, project-docs, prioritization, product-requirements, roadmap-planning, session-notes, and more.
SME Trio PatternPartnership evaluations spawn 3 parallel subagents (regulatory + clinical + product-tech) that synthesize into a scored recommendation. Powerful orchestration pattern.
Memory SystemCross-session persistent memory: preferences.md, learnings.md (curated), learnings-staging.md (pending). Domain agents write findings before returning.
Smart-search IntegrationVault-first research: meeting prep, daily briefing, and other skills check Obsidian vault via smart-search MCP before falling back to web search.
54 Structural TestsPre-commit validation of manifest, agents, skills, persona, rules, and cross-references. Prevents silent drift.

Current Limitations

No Wiki/Compilation SkillsJeeves has no skills for generating companion MDs, extracting entities, building cross-linked knowledge pages, or maintaining an index. Key Gap
No Vault Structure AwarenessJeeves doesn't understand your PARA structure. It can search files but doesn't know "this folder is a Project" vs "this folder is Research." Key Gap
No File Coverage AuditNo way to know which raw files have associated notes and which are "orphaned" (no knowledge captured from them). Key Gap
0 Triggering TestsNo automated tests for whether the right agent/skill gets activated for a given query. Structural tests only validate file formats.
0 Behavioral TestsEvaluation framework designed but not executed. No baselines for output quality or token efficiency.
Orphan Skills5 skills (clinical-expertise, obsidian-knowledge, regulatory-intelligence, executive-communication, curate-memory) have no command entry point, rely on implicit triggering.

How They Connect Today

# Current integration (simplified): You ask Claude Code a question (e.g., "Prepare me for the FDA meeting tomorrow") ↓ Jeeves routes to meeting-prep skill ↓ meeting-prep triggers "vault-first research": ↓ Tier 1: Calls smart-search MCP → knowledge_search("FDA meeting attendees") ↓ smart-search searches its index of raw documents (PDF chunks, DOCX chunks, etc.) ↓ Returns: matching chunks with file paths and relevance scores ↓ Tier 2 (fallback): Grep/Glob across vault if smart-search unavailable ↓ Jeeves synthesizes results into meeting prep brief # The problem: smart-search returns raw document chunks, not synthesized knowledge. # There are no entity pages for attendees, no concept pages for FDA pathways. # Every question requires re-synthesizing from raw sources every time.

What Changes with the Knowledge Base System

# Future integration (with wiki layer): You ask Claude Code: "Prepare me for the FDA meeting tomorrow" ↓ Jeeves routes to meeting-prep skill ↓ meeting-prep triggers "wiki-first research": ↓ Tier 0: Check _knowledge/ entity pages: → _knowledge/entities/fda-510k.md — pre-synthesized regulatory pathway knowledge → _knowledge/entities/dr-smith.md — attendee background from past meetings → _knowledge/entities/product-cams.md — everything we know about the product # These pages already contain cross-referenced, synthesized knowledge! ↓ Tier 1: Semantic search via smart-search/QMD across all MD notes → Searches _wiki/ companion MDs + _knowledge/ pages + your notes → Finds specific meeting transcripts, regulatory documents, clinical evidence ↓ Tier 2 (fallback): Grep/Glob ↓ Jeeves synthesizes a much richer meeting prep brief because: - Entity pages provide pre-synthesized context (no re-reading raw documents) - Cross-links surface connections you might have missed - Source attribution traces back to specific documents # The knowledge graph means answers are faster, richer, and more connected.
