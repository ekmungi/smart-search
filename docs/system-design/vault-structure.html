Vault Structure

How the wiki layer integrates with your existing PARA-organized Obsidian vault without disrupting your workflow.

Design Principle: Respect What Exists

Your PARA structure stays exactly as-is. The wiki layer adds two new things: (1) _wiki/ subfolders inside existing folders for auto-generated companion MDs, and (2) a vault-level _knowledge/ folder for cross-cutting entity and concept pages. Your existing notes, your folder organization, your file locations — none of these change. The underscore prefix (_wiki/, _knowledge/) visually separates generated content from your own work.

What the Search Engine Indexes

Only Markdown files get indexed by the search engine. The raw binary files (PDF, DOCX, PPTX, XLSX) are not indexed directly. Instead, their knowledge is captured in either your handwritten notes or auto-generated companion MDs. The search engine only needs to index three types of MD files:

1

Your Notes

Meeting notes, literature notes, project notes, congress notes, and all other MD files you've written. These are spread across Projects/, Areas/, and Resources/ folders. They are the primary knowledge source and are indexed directly.

2

Companion MDs

Auto-generated summaries in _wiki/ subfolders for raw files that don't have your handwritten notes. These fill the gaps so every file is represented in the search index as Markdown text.

3

Knowledge Pages

Entity and concept pages in _knowledge/. These are the highest-value search targets because they contain pre-synthesized, cross-linked knowledge from multiple sources. Searching here first (Tier 0) gives the best answers.

Key Design Rules

Rule 1: Never duplicate user notes If you've written a meeting note that links to a transcript, the system uses your note as the knowledge source for that file. It does NOT create a second summary in _wiki/. Your note is always preferred over auto-generated content.
Rule 2: _wiki/ is for gaps only Auto-generated companion MDs only appear for raw files that have no associated user note. If you later write a note for a file, the companion MD in _wiki/ can be archived or deleted (the wiki-lint skill helps identify these).
Rule 3: _knowledge/ is cross-cutting Entity and concept pages aggregate knowledge from ALL sources across the entire vault. A single entity page (e.g., fda-510k.md) might draw from regulatory guidance in Resources/, meeting transcripts, project planning docs, and your own notes. This cross-pollination is the primary value of the knowledge graph.
Rule 4: Archives are frozen When a project or area is archived, its _wiki/ subfolder moves with it but is no longer updated. Entity pages in _knowledge/ retain their historical references but are flagged as "from archived source" to indicate potential staleness.
Rule 5: Raw files are immutable The system never modifies your original files. PDFs stay as PDFs. DOCX stays as DOCX. Only new files are created (companion MDs, knowledge pages).
Rule 6: Provenance is always traceable Every companion MD has source_path: frontmatter pointing to its raw file. Every knowledge page cites specific source notes with Obsidian wiki-links. You can always trace a claim back to the original document.
