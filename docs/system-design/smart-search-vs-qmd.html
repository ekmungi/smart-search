Smart-search vs QMD

A critical, honest comparison to decide whether to continue developing smart-search or adopt QMD as the search layer.

The Key Question

In the new architecture where every raw file has a companion MD, and only Markdown files get indexed, smart-search's biggest differentiator (14-format conversion via MarkItDown) becomes irrelevant for the search layer. The conversion capability moves to the compilation layer (Jeeves wiki-compile skill). Both smart-search and QMD can index Markdown files. So the question becomes: which one does Markdown search better?

The Honest Assessment

Arguments for Keeping Smart-search

1. Already built and integrated. Smart-search works today. It's deeply integrated with Jeeves (11 MCP tools, dedicated skill). Switching to QMD requires re-integration work: new MCP configuration, updated Jeeves skills, testing.

2. Desktop app. The Tauri desktop app provides a visual interface for managing folders, viewing indexing status, and quick searching. QMD has no GUI. If you value the visual management experience, this matters.

3. File watching. Smart-search automatically detects and re-indexes changed files. QMD requires manual re-indexing (qmd embed). In the new architecture, the file watcher serves double duty: triggering both search re-indexing and wiki compilation.

4. You control it. Full ownership means you can add features, fix bugs, and customize behavior without waiting for upstream changes. Smart-search is exactly what you need because you shaped it.

5. MarkItDown integration. Even though format conversion moves to the compilation layer, having MarkItDown already integrated in smart-search means the wiki-compile skill can leverage smart-search's existing read_note MCP tool for file conversion.

Arguments for Switching to QMD

1. Superior search quality. QMD has query expansion, LLM reranking (not just a tiny cross-encoder), and position-aware blending. For a knowledge base system where search quality directly impacts the quality of Jeeves agent answers, this matters significantly.

2. Context system. QMD's ability to attach metadata to collections/paths is a natural fit for PARA. Your Jeeves agents would receive search results with context like "this result comes from a regulatory resource" or "this is from an active project document." Smart-search has nothing comparable.

3. Maintenance burden. You are a scientist, not a software developer. Maintaining a complex Python codebase with FastAPI, LanceDB, ONNX, Tauri, and 425 tests is significant overhead. QMD is a dependency you use, not code you maintain. Your time is better spent on the compilation layer (wiki skills) and your actual PM work.

4. Purpose-built for this pattern. QMD was designed to be the search layer for markdown knowledge bases. It's the tool Karpathy recommends. It solves exactly the problem you have, and it's optimized for agentic workflows (LLM agents searching your knowledge base).

5. Simpler storage. Single SQLite file vs. LanceDB + SQLite. Easier to backup, migrate, debug.

Recommended Path Forward

Phase 1: Keep Smart-search (Now)

Don't change the search layer while building the compilation layer. Smart-search works today. Use it. Focus your development effort on the high-value, novel work: the wiki-crawl, wiki-compile, wiki-ingest, wiki-query, and wiki-lint skills. These are the skills that make the Karpathy pattern work. The search engine underneath is a replaceable commodity — the compilation layer is the differentiator.

↓

Phase 2: Evaluate QMD (After compilation layer works)

Once the wiki skills are working and you have a meaningful set of companion MDs and knowledge pages, install QMD alongside smart-search. Index the same content in both. Run 20-30 representative queries through both engines and compare result quality. Pay special attention to:
• Does QMD's query expansion find things smart-search misses?
• Does the context system improve Jeeves agent decisions?
• Is the LLM reranker noticeably better than TinyBERT?
This is an empirical question that should be answered with your actual data, not theoretical analysis.

↓

Phase 3: Decision (Based on evidence)

If QMD is clearly better: Migrate search to QMD. Update Jeeves's smart-search skill to proxy to QMD instead. Keep smart-search's MarkItDown + Watchdog capabilities in a lightweight utility for the compilation layer. Stop maintaining the search engine.

If QMD is comparable: Consider maintenance burden. If smart-search is consuming significant development time, switch to QMD anyway for the time savings. If it's stable and low-maintenance, keep it.

If smart-search is better for your use case: Keep it. Consider adopting specific QMD ideas (context system, query expansion) into smart-search.

What Smart-search Built That Remains Valuable

Regardless of the search engine decision, smart-search has produced valuable capabilities that the new system needs:

MarkItDown IntegrationThe ability to convert 14 file formats to Markdown text is essential for the compilation layer. The wiki-compile skill needs this. Whether it lives in smart-search or as a standalone Python utility, this code has direct value.
File Watching (Watchdog)The file change detection system with debouncing and per-folder management is needed to trigger wiki compilation automatically. This can be extracted from smart-search if you migrate to QMD.
MCP Server PatternThe FastMCP server design (thin proxy to HTTP backend) is a proven pattern that can be reused for new MCP tools (e.g., a wiki-management MCP).
Chunking LogicThe heading-based Markdown chunker, while simpler than QMD's, is well-tested and could be used by the compilation layer for splitting large documents before LLM processing.
Research FindingsThe IR research documents (hybrid search impact, cross-encoder benchmarks, Matryoshka dimension analysis) inform design decisions regardless of which search engine you use.
