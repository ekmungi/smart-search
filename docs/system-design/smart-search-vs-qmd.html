Smart-search vs QMD

Decision: keep smart-search. The 30x RAM advantage and local-first, no-GPU design make it the right engine for this system.

The Deciding Factor

Local-first, no GPU. Smart-search runs at ~130 MB RAM with CPU-only inference. QMD pulls in node-llama-cpp with a 0.6B reranker and a 1.7B query-expansion model, landing at ~4–6 GB RAM and expecting GPU acceleration for acceptable latency. On a standard laptop without a discrete GPU, smart-search is the only option that stays responsive. This is a 30x difference in memory footprint — and it is the deciding factor.

In the new architecture where every raw file has a companion MD and only Markdown files get indexed, smart-search's biggest differentiator (14-format conversion via MarkItDown) becomes irrelevant for the search layer. The conversion capability moves to the compilation layer (Jeeves wiki-compile skill). Both smart-search and QMD can index Markdown files. So the question was: which one does Markdown search better? QMD has more sophisticated retrieval techniques — but they come at a resource cost that disqualifies it for a local-first, no-GPU workflow. The right move is to keep smart-search and selectively port the best QMD ideas.

Learnings to Port from QMD → Smart-search

QMD is not the right engine for this system, but it contains three innovations worth adopting in smart-search:

1. Query Expansion (Lighter Version)

QMD uses a fine-tuned 1.7B model to generate query variants before search. This dramatically improves recall when the user's phrasing doesn't match the document text.

Port to smart-search: Add a lightweight query expansion step that calls an LLM API (e.g., Claude or a local small model) to generate 2–3 reformulations of the query. Run all variants through the existing BM25 + vector pipeline and merge results. This avoids loading a 1.7B model locally while capturing most of the recall benefit. Could also use a tiny local model (<100 MB) for offline scenarios.

2. Collection / Path Context

QMD attaches folder-path metadata to results, so an LLM agent receiving search results knows whether a document came from Projects/, Resources/, or Archives/ — and can prioritize accordingly.

Port to smart-search: Smart-search already stores folder info in its metadata. The change is small and high-value: include the folder path and an optional context string in every MCP search result. For PARA, attach labels like "Active project" or "Reference material" at the folder level. This is a metadata-formatting change, not an architecture change.

3. Position-Aware Blending

QMD weights results based on position in both BM25 and vector rankings: top-ranked results trust the initial RRF score more, while lower-ranked results rely more on the reranker. This adaptive blending outperforms flat RRF.

Port to smart-search: Smart-search currently uses standard RRF with k=60 and a single reranking pass. The improvement would be to tune the RRF constant and blending weights per document type (e.g., short meeting notes vs. long reference documents). Could also add a simple position decay so the reranker has more influence on borderline results where the initial ranking is uncertain.

Recommended Path Forward

Phase 1: Keep Smart-search, Focus on Compilation (Now)

The decision is made: smart-search is the search engine. Its ~130 MB footprint, CPU-only inference, and deep Jeeves integration make it the right fit for a local-first, no-GPU system. Focus development effort on the high-value, novel work: the wiki-crawl, wiki-compile, wiki-ingest, wiki-query, and wiki-lint skills. These are the skills that make the Karpathy pattern work. The search engine underneath is a solved problem — the compilation layer is the differentiator.

↓

Phase 2: Port QMD Innovations as Future Improvements

Once the compilation layer is stable and producing companion MDs at scale, selectively port the three QMD innovations into smart-search:
• Query expansion via LLM API or small local model (highest expected impact on recall).
• Path context in MCP results (smallest change, immediate value for Jeeves agents).
• Position-aware blending / RRF tuning (requires experimentation with real queries).

These are incremental improvements to an already-working system — not a rewrite. Prioritize path context first (a few hours of work), then query expansion (a few days), then blending tuning (ongoing experimentation).

What Smart-search Built That Remains Valuable

Regardless of the search engine decision, smart-search has produced valuable capabilities that the new system needs:

MarkItDown IntegrationThe ability to convert 14 file formats to Markdown text is essential for the compilation layer. The wiki-compile skill needs this. Whether it lives in smart-search or as a standalone Python utility, this code has direct value.
File Watching (Watchdog)The file change detection system with debouncing and per-folder management is needed to trigger wiki compilation automatically. This can be extracted from smart-search if you migrate to QMD.
MCP Server PatternThe FastMCP server design (thin proxy to HTTP backend) is a proven pattern that can be reused for new MCP tools (e.g., a wiki-management MCP).
Chunking LogicThe heading-based Markdown chunker, while simpler than QMD's, is well-tested and could be used by the compilation layer for splitting large documents before LLM processing.
Research FindingsThe IR research documents (hybrid search impact, cross-encoder benchmarks, Matryoshka dimension analysis) inform design decisions regardless of which search engine you use.
