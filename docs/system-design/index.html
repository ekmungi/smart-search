Personal Knowledge Base System

Karpathy's LLM Wiki pattern applied to a PARA-organized Obsidian vault, powered by Jeeves + Smart-search. Pilot: Meetings folder.

Karpathy's LLM Wiki Pattern

Key Insight: "Humans abandon wikis because the maintenance burden grows faster than the value. LLMs don't get bored, don't forget to update a cross-reference, and can touch 15 files in one pass."

Instead of RAG (re-searching raw documents every time), the LLM does synthesis, cross-referencing, and contradiction detection at ingest time. The wiki becomes a persistent, compounding artifact where intellectual work is pre-computed. Queries are fast and high-quality because connections already exist.

Human role: curate sources, direct analysis, ask questions, think about meaning.
LLM role: summarize, cross-reference, file, maintain consistency.

Three Layers

Layer 1 — Raw Sources

Your Files (Immutable)

Meeting transcripts, PDFs, DOCX, PPTX, XLSX, images, research papers. These live in your PARA vault and are never modified by the system. They are the source of truth.

↓
Layer 2 — The Wiki

LLM-Generated Knowledge (_knowledge/)

The compounding artifact. Entity pages, concept pages, cross-references, and discussion timelines — all generated and maintained by the LLM. Lives in a vault-level _knowledge/ folder. Each ingested source touches 10-15 wiki pages. Two special files anchor everything:

index.md — Master Catalog

Every wiki page listed with one-line summary, organized by category. The LLM reads this FIRST on every query to know what knowledge exists. Updated on every ingest.

log.md — Chronological Record

Append-only. Each entry: ## [YYYY-MM-DD] ingest | Source Title. Parseable by simple tools. Never edited, only appended. Full audit trail.

↓
Layer 3 — The Schema

Conventions & Skills

Configuration defining wiki structure, page templates, cross-link syntax, and workflows. This is _knowledge/SCHEMA.md plus the Jeeves skill definitions (vault-audit, process-meeting, task-creator). The schema ensures consistency across the entire knowledge base.

Three Operations

Ingest

New source arrives (e.g., meeting transcript). The process-meeting skill reads it, enhances the meeting note with structured frontmatter and wiki-links, extracts entities/concepts into _knowledge/ pages, updates index.md, appends to log.md. Touches 10-15 files per source. Also discovers related meetings via semantic search and entity co-occurrence.

Query

User asks a question. LLM reads index.md first to locate relevant pages, then checks entity/concept pages, then falls back to smart-search semantic search and Grep/Glob. Answers can become new wiki pages — queries compound into knowledge.

Lint

Periodic health check: contradictions across pages, stale claims, orphan pages (no incoming links), missing cross-references, data gaps. Not built in the pilot — designed for later.

Pilot: Meetings Folder First

We start with one folder, not the whole vault. The Meetings folder is ideal because: meeting notes already have natural structure (date, attendees, topics, decisions, action items), transcripts are the raw files that need processing, stakeholder files in Resources link naturally to meeting attendees, and the existing meeting-prep skill in Jeeves immediately benefits from structured, cross-linked meeting knowledge.

Step 1 is vault-audit, not building. We've never seen the actual vault structure. Before building anything, the vault-audit skill discovers folder names, file formats, existing frontmatter conventions, and the transcription tool's output format. Everything else adapts to what the audit reveals.

Step 1: vault-audit

Discover the real vault structure. Map folder names, file types, frontmatter conventions, transcription tool output format, stakeholder files. Produces a structured report. Build this first.

↓

Step 2: process-meeting (Karpathy's "Ingest")

Takes a meeting (notes MD + transcript) and produces three outputs: (1) Enhanced meeting note with structured frontmatter, stakeholder wiki-links, and related meetings. (2) Tasks/action items linked to projects and assignees. (3) Knowledge entries in _knowledge/ (entity pages, index, log).

↓

Step 3: Update existing skills

Enhance meeting-prep with structured frontmatter queries and _knowledge/ entity lookups. Update obsidian-knowledge to read index.md first (Tier 0). Add task-creator for standalone task management.

Search Layer: Smart-search (Decision: Keep)

Smart-search stays. RAM footprint: ~130MB vs QMD's ~4-6GB (30x difference). On a no-GPU laptop, smart-search is the clear local-first choice. QMD's search quality innovations (query expansion, LLM reranking, collection context) are being studied for future smart-search improvements. See detailed comparison.
